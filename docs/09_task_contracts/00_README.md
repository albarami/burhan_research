# Burhān — Task Contracts (docs/09_task_contracts/00_README.md)

**Status:** For review · **Governed by:** `08_BUILD_SPEC.md` (source of truth), `10_PROJECT_PLAN.md` (order & gates), `15_ENGINEERING_STANDARDS.md` §6 (workflow), `CLAUDE.md` / `AGENTS.md` (roles).

A task contract is the unit of work: one contract → one branch → one PR → one binary verdict. Contracts **restate** their module's scope, traces, and acceptance tests from the build spec verbatim and add delivery detail only — a contract can never introduce scope the build spec lacks. These files were generated mechanically from `08_BUILD_SPEC.md` with an ownership check proving every acceptance test (75/75) is owned by exactly one contract.

## Lifecycle

`DRAFT` (authored) → `ISSUED` (Codex issues; dependencies all APPROVED) → `IN_PROGRESS` (Claude Code on branch) → `APPROVED` (verdict recorded in `SIGNOFFS.md`). REJECT keeps the contract `IN_PROGRESS` with the fix list attached to the PR. State lives in the ledger, not by editing contract files.

**TC-18** is a researcher-authored engine follow-up (like TC-15/TC-16, not part of the mechanical TC-01..TC-14 / 75-AT generation): an M6 ingest-crosswalk remediation surfaced by the live validation run. Against a real 3-header Qualtrics export the V6 zero-orphan crosswalk halts — `build_crosswalk` defaulted `header_rows` to 1 and scanned row 0 (Qualtrics question IDs) instead of row 1 (the question-text row carrying the embedded item codes), so every modeled item read as `missing_items` (FR-104); the same embedded-code mismatch would next block role accounting for the demographics / ignored / metadata codes. TC-18 makes the crosswalk resolve **all** declared codes (modeled and non-modeled) from the embedded-code header row, keyed to the row-0 identifiers, with the header mechanism — (a) Node A extracts `data.header_rows`, or (b) the crosswalk detects/validates the Qualtrics dialect — chosen in PLAN v1 from the actual schema and code (the schema carries only `data.header_rows`, with no code-row-index field), and **halting, never silently defaulting**, on an ambiguous multi-header structure. It preserves V6 strictness (a genuine orphan still halts), tests against a fixture modeled on the real 3-header structure (never the researcher's raw data), and carries acceptance tests AT-M18-1..6. It must be APPROVED before the M6 validation run completes Gate 1.

**TC-19** is a researcher-authored engine defect fix (like TC-15/TC-16/TC-18, not part of the mechanical TC-01..TC-14 / 75-AT generation): an M6 Gate-1 R-worker invocation fix surfaced by the live validation run. With TC-18's crosswalk merged, the DBA `--live --confirm` run cleared contract → ingest → Gate 1, then halted at the first R statistics worker (`power`) with `RENV_DRIFT: renv status not synchronized for workers/r` (NFR-102). A read-only five-context diagnosis proved this a **false positive**: `RWorker.call` spawns `subprocess.run` for `Rscript harness.R …` with no `cwd`/`env`, so from the burhan process's working directory (repo root, which has no `.Rprofile`) the `workers/r` project renv never activates, `.libPaths()` stays on the system library, and `harness.R`'s `renv::status(project=workers/r)` check compares the lockfile against the wrong library. The `workers/r` renv project is itself consistent and git-clean, so `renv::restore()` is the wrong fix. TC-19 runs the worker subprocess in the `workers/r` project context — setting `cwd` so the project `.Rprofile` → `renv/activate.R` sets `.libPaths()` before the NFR-102 check — and normalizes `self._workers_dir` to an absolute path for cwd/path safety, while keeping the guard fully functional against *real* drift. Scope is `src/burhan/core/rworker.py` + its unit tests only; no change to worker logic, `harness.R` semantics, `renv.lock`, the renv library, study data, the reference file, or pipeline behavior. It carries acceptance tests AT-M19-1..5 and must be APPROVED and merged before the M6 validation run resumes past `power`.

**TC-20** is a researcher-authored engine fix + governed study-source correction (like TC-15/TC-16/TC-18/TC-19, not part of the mechanical TC-01..TC-14 / 75-AT generation): an M6 prep-stage demographic-crosswalk remediation surfaced by the live validation run. With TC-19 merged, the DBA `--live --confirm` run cleared contract → ingest → Gate 1 → `power`, then halted at **prep** with `declared column resolves to more than one export column (FR-103/104)`: the demographic `D3` (`column_hint: D3`) matched both its Qualtrics selected-choice column `Q44` and the paired `Q44_5_TEXT` "Other (please specify)" free-text sidecar, whose row-1 question text embeds the same `D3` label (`D5`/`D7` share the pattern). Read-only inspection of the raw 3-header export resolved the full demographic picture and two distinct problems: (1) **selected-choice + text-sidecar ambiguity** — the crosswalk must bind each affected demographic to its selected-choice column and account the paired `..._N_TEXT` sidecar as an ignored column, without weakening the real-ambiguity halt; and (2) **source-contract errors** — `D4` (nationality, `Q45`), whose question text carries no `D4` label so `column_hint: D4` never matches, and a phantom `D10` the instrument never exported (the survey skips `D9`→`D11`). Researcher decisions are recorded in the contract: `D3`→`Q44`, `D4`→`Q45`, `D5`→`Q46`, `D7`→`Q48`; the four `_TEXT` sidecars (`Q44_5_TEXT`, `Q45_2_TEXT`, `Q46_7_TEXT`, `Q48_9_TEXT`) accounted as ignored; `D4` kept as a descriptive (non-modeled) demographic; `D10` removed. Scope is `src/burhan/contract/crosswalk.py` + its unit tests, plus the governed study-source correction for `D4`/`D10` (PLAN v1 decides whether that requires a re-extraction or the existing `study_config.yaml` stays valid for a direct `--confirm`); the 11-construct model / 45 items / 9 hypotheses are untouched, and the V6 zero-orphan / FR-103/104 real-ambiguity guards stay intact (AT-M20-2, AT-M20-5). It tests against a fixture modeled on the real Qualtrics demographic structure (never the raw data), carries acceptance tests AT-M20-1..6, and must be APPROVED before the M6 validation run completes prep.

## Change control

Contract text is researcher-owned. If implementation reveals a contract defect (wrong criterion, untestable clause, conflict with a governed document), the lane STOPS and escalates (AGENTS.md); the researcher amends the build spec first, contracts are regenerated or edited to match, and the hash trail records it. Codex and Claude Code never patch a contract to fit the code.
