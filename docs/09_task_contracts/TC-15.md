## Why this contract exists

The build spec specified the orchestrator's behavior (M04) and every stage's internals (M05–M14) but never created a work item to **register the fourteen stages into the orchestrator's DAG**. Consequently `burhan run` correctly refuses with all stages unregistered, and certification C4 (IT-1..IT-4) cannot pass. This contract closes exactly that gap — integration, not new analysis.

## Objective & Scope

Wire the existing, individually-certified Stage-1A modules into the existing orchestrator so the full pipeline executes end-to-end under deterministic stubbed LLM nodes, satisfying IT-1..IT-3.

**In scope:**
- Register the Stage-1A stages (S0 INGEST → S8 ROBUSTNESS, plus the Gate-1 node) into the orchestrator's stage registry, using the `Stage` protocol already defined in M04.
- Assemble each stage as a thin adapter over its **already-built** module (`prep/*`, `stats/*`, `verify/*`, `contract/*`, `review/node_c`, governance, playbook/compliance, provenance) — calling existing functions, adding no new statistical behavior.
- Deterministic **stubbed** Node A / Node C fixtures (canned, schema-valid responses; no provider calls) sufficient to drive a golden study through the pipeline.
- The production stage registry returns the wired stages (replacing the empty `{}`).
- Tests delivering IT-1, IT-2, IT-3 (see acceptance).

**Explicitly out of scope:**
- Any change to a stage module's statistical behavior, thresholds, or outputs. Adapters call existing code; they do not reimplement it. If an adapter cannot be built without changing a module's behavior, STOP and escalate — that is a defect to surface, not to patch.
- Stage-1B stages (S9 NARRATE, S10 PACKAGE) — those remain with TC-13/TC-14. This contract wires Stage-1A only, which is what C4/IT-1..IT-3 exercise on the certified workstation.
- Governed-document edits (schemas, playbook, policy, registry, concept) — none required; if one appears necessary, escalate.
- New dependencies.

## Deliverables

- `src/burhan/` orchestrator stage-registry wiring + the thin stage adapters (under the orchestrator/stages area established in M04).
- `tests/integration/` (authorized here and by certification-plan §6): the IT-1..IT-3 harness with stubbed-node fixtures.
- Stubbed Node A/C fixtures under `tests/fixtures/`.

## Acceptance Tests (binding)

- **AT-M15-1 (IT-1 dry run):** a golden study runs end-to-end to terminal state `COMPLETED` under stubbed nodes; `METHOD_COMPLIANCE_CHECKLIST.md` covers PB-01..PB-19 with no step unaccounted; every stage recorded byte-consistently in the run archive.
- **AT-M15-2 (IT-2 rerun identity):** `burhan rerun` on the sealed AT-M15-1 run regenerates byte-identical artifacts (NFR-101); a planted nondeterminism in a stub is caught by the identity assertion.
- **AT-M15-3 (IT-3 boundary):** an under-powered golden fixture drives the pipeline to `METHOD_ADVISORY` via governance and terminal state `COMPLETED_TO_BOUNDARY`, producing a defensible-scope package up to the advisory boundary.
- **AT-M15-4 (registry non-empty):** the production stage registry returns exactly the Stage-1A stages in DAG order; an independent probe confirms `burhan run` no longer exits 10 for a well-formed golden study.
- **AT-M15-5 (no behavior drift):** the full pre-existing unit + golden + benchmark suites remain green unchanged — proving the wiring added no statistical behavior (adapters over existing code only).
