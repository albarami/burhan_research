## Why this contract exists

M05/TC-05 specified Qualtrics-dialect parsing of multi-header exports with a column→item crosswalk from embedded item codes. The live DBA run surfaced two real gaps in that implementation against an actual 3-header Qualtrics CSV:

1. **Primary — header-row selection.** The export has three header rows: row 0 = generic Qualtrics question IDs (`Q5_1`, `Q8_1`…), row 1 = question text with **embedded item codes** (`"Internal Expertise - R3. …"`), row 2 = ImportId JSON. `build_crosswalk` derives `header_rows = config.data.header_rows if not None else 1`, and Node A did not extract `header_rows`, so it defaulted to 1 → the item-code matcher scanned row 0 (no codes) → FR-104 "structural mismatch," `missing_items = all modeled items`. The codes are present in row 1 (verified: R3, R9, A6, C25, C43, T6, T16 all found in row-1 text; zero in rows 0 and 2).

2. **Secondary — role accounting from embedded codes.** `_account_roles` matches `demographics.column_hint`, `metadata_columns`, and `ignored_item_columns` by **exact identity against row 0** column names. But in this export the demographic and out-of-model codes (D1…, R1, R2, the ignored C/T items) also live embedded in row-1 question text, not as literal row-0 identifiers. So even after the header fix, role accounting would halt next on a declared column the export "lacks."

Both are the same underlying defect: the crosswalk must resolve **all** declared codes (modeled items, demographics, ignored items, metadata) from the embedded-code header row of a multi-header Qualtrics export, using the row-0 IDs as the stable column keys.

## Objective & Scope

Make the ingest-crosswalk correctly parse real multi-header Qualtrics exports so every declared code (modeled and non-modeled) resolves to its export column, and V6 zero-orphan accounting passes for the DBA study without reshaping the raw data.

**In scope:**
- **Header-row handling.** Establish the authoritative number of header rows and which row carries embedded codes. Mechanism is chosen in PLAN v1 (see "Researcher decision" below): either (a) Node A extracts `data.header_rows` from the sources; or (b) the crosswalk detects/validates the header block for the Qualtrics dialect. Note the schema currently carries only `data.header_rows` and no code-row-index field. The chosen mechanism must be explicit and tested, and must **halt (not silently default)** when the header structure is ambiguous or unstated for a multi-header export.
- **Embedded-code resolution for ALL roles.** `_match_items` and `_account_roles` must resolve modeled item codes AND non-modeled declared codes (demographics, ignored items, metadata) from the embedded-code header row, keyed to the row-0 column identifiers, using the existing whole-token regex `(?<![A-Za-z0-9_]){code}(?![A-Za-z0-9_])`. Role accounting must not require declared codes to appear as literal row-0 identifiers when the export is multi-header with embedded codes.
- **Zero-orphan integrity preserved.** After resolution, every export column must still resolve to exactly one role, and a genuinely undeclared column must still be flagged an orphan (V6 unchanged in strictness — only the *matching mechanism* is corrected).
- Tests using a **fixture** modeled on the real 3-header Qualtrics structure (row 0 IDs, row 1 embedded codes, row 2 ImportId) — not the researcher's raw data in the repo.

**Explicitly out of scope:**
- No weakening of V6/zero-orphan or any validator — a real orphan must still halt.
- No hand-editing of the DBA `study_config.yaml` or the raw CSV.
- No change to statistical modules, certification, or other governed docs beyond what the chosen header mechanism requires (and if it touches the Node A prompt, that stays within prompt+tests and must keep the prompt-coverage meta-test green).
- The `_resolve_inputs` CSV-selection behavior (`sorted(glob("*.csv"))[0]` vs `config.data.file`) is a **separate** known item — note it, do not fix it here unless it blocks the fixture test; if it does, the minimal fix is to select by `config.data.file` when present.

## Researcher decision required (pick before implementation)

**How is `header_rows` / the code-bearing row established?**
- **Mechanism (a): Node A extracts it.** Cleaner long-term (the contract fully describes the export), but reopens the Node A prompt. RECOMMENDED if the sources state the 3-row structure.
- **Mechanism (b): crosswalk detects/validates the Qualtrics dialect.** Keeps the fix inside ingest, no prompt change, but encodes dialect knowledge in the parser.

No default is pre-bound. PLAN v1 must recommend the mechanism with explicit lower-risk reasoning from the ACTUAL schema and code, noting: the schema supports only `data.header_rows` (study_config.schema.yaml) — there is **no** code-row-index field — so Mechanism (a) would require both a live Node A re-extraction and possibly a schema addition, whereas Mechanism (b) keeps the fix inside the crosswalk, avoids prompt/schema churn, and likely preserves the existing `study_config.yaml` (no new Node A token). PLAN v1 weighs these and decides.

## Acceptance Tests (binding)

- **AT-M18-1 (modeled items resolve):** on a 3-header Qualtrics fixture (row 0 IDs, row 1 embedded codes, row 2 ImportId) with `header_rows` established by the chosen mechanism, every modeled item code resolves to its column; `missing_items` is empty.
- **AT-M18-2 (all roles resolve):** demographics, ignored items, and metadata declared by embedded code also resolve; role accounting completes with no declared-column-lacking halt.
- **AT-M18-3 (zero-orphan preserved):** every column resolves to exactly one role; a fixture with a genuinely undeclared column still halts as an orphan (V6 strictness intact).
- **AT-M18-4 (ambiguity halts, no silent default):** a multi-header fixture with `header_rows` unstated/ambiguous halts typed (does not silently assume 1).
- **AT-M18-5 (single-header still works):** a plain single-header CSV fixture (codes as literal row-0 headers) still resolves correctly — no regression for non-Qualtrics exports.
- **AT-M18-6 (no drift):** the full prior unit + golden + benchmark + integration + prompt-coverage suites stay green; if the Node A prompt changed, the prompt-coverage meta-test and DF-001-tracked coverage remain intact.
