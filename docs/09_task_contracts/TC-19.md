## Why this contract exists

The DBA `--live --confirm` run passed contract → ingest → Gate 1, then halted at the first R statistics worker with `RENV_DRIFT: renv status not synchronized for workers/r` (NFR-102). Read-only diagnosis (5-context comparison) proved this is a **false-positive drift caused by an invocation defect**, not real library drift and not a data/config/TC-18 issue:

- `RWorker.call()` builds `argv = [Rscript, harness.R, worker.R, in.json, out.json]` and calls `subprocess.run(...)` **without `cwd` or `env`** (rworker.py:99-109). The subprocess therefore inherits the burhan process's working directory (repo root, which has no `.Rprofile`) and inherited env (`RENV_PROJECT`/`R_LIBS`/`R_LIBS_USER` all unset).
- `harness.R` calls `renv::status(project = dirname(worker_path))` **without first sourcing renv activation** (harness.R:14-19), so `.libPaths()` stays on the system library.
- `renv::status` then compares the `workers/r` lockfile against the **system** library → reports not-synchronized → NFR-102 halt.

Proof the environment is actually consistent: with the worker's own project context active, `renv::status(project=workers/r)` returns **"No issues found"** in two independent reproductions (cwd=`workers/r`, which auto-sources `.Rprofile`→`activate.R`; and explicit `source(activate.R)` + `RENV_PROJECT=workers/r`). `workers/r/renv.lock` and `workers/r/renv/` are git-clean from the prior round. A `renv::restore()` would be a no-op at best, a mass re-download at worst — the wrong fix.

## Objective & Scope

Run the R worker subprocess in the `workers/r` project context so the project `.Rprofile` activates renv (setting `.libPaths()` to the project library) **before** `harness.R` performs its NFR-102 `renv::status` check — clearing the false drift while keeping the guard fully functional against *real* drift.

**In scope:**
- Set `cwd` to the `workers/r` project directory on the `subprocess.run(...)` call in `RWorker.call()`. This causes Rscript to auto-source the project `.Rprofile` → `renv/activate.R` → correct `.libPaths()`, exactly as renv is designed to bootstrap.
- **Path-safety:** normalize `self._workers_dir` to an absolute path in `RWorker.__init__` (it is currently `Path(__file__).resolve()`-based for the default, but a caller-injected `workers_dir` is not normalized). All four Rscript path arguments (harness, worker, input, output) are already absolute (`self._workers_dir` resolved; input/output derive from the absolute run dir), so a `cwd` change cannot break them — this contract confirms and, if needed, enforces that.
- Unit + regression tests (below).

**Explicitly out of scope:**
- No change to statistical worker logic, `harness.R` semantics, or the NFR-102 check's *meaning* — real drift must still halt.
- No change to `renv.lock`, the renv library, study data, the reference file, or any pipeline behavior.
- No `renv::restore()`, `renv::snapshot()`, or package install.
- No governed-document edits beyond adding this contract + its index entries.
- Alternative mechanisms (sourcing activate.R inside harness.R, injecting `RENV_PROJECT`) are **rejected** by the diagnosis: sourcing alone activates the wrong project and triggers a download unless also given `RENV_PROJECT` (and edits governed harness.R); `RENV_PROJECT` alone never activates. The `cwd` fix is the minimal, correct one and edits no governed R file.

## Requirement Traces

NFR-102 (environment integrity / reproducibility — the guard being unblocked, and which must remain functional), AD-02 (stateless file-based R workers), FR-1204 (halt classes). The fix restores the intended TC-04 worker contract: the worker runs with the project renv active.

## Acceptance Tests (binding)

- **AT-M19-1 (cwd set):** `RWorker.call()` invokes `subprocess.run` with `cwd` equal to the `workers/r` project directory — asserted directly (e.g. by capturing the `subprocess.run` call args).
- **AT-M19-2 (drift clears with cwd, halts without):** a regression test reproduces the false `RENV_DRIFT` when `cwd` is absent (system-library context) and proves it clears (worker reaches computation / no NFR-102 halt) when `cwd` is set to the project — demonstrating the fix addresses the exact failure.
- **AT-M19-3 (real drift still halts):** a genuinely unsynchronized renv context (real drift) still produces the `RENV_DRIFT` stderr mapped to `IntegrityHalt` — the guard is not weakened, only given the correct context.
- **AT-M19-4 (path safety):** `self._workers_dir` is absolute after `__init__` for both the default and a caller-injected relative `workers_dir`; all four Rscript path args remain absolute so the `cwd` change cannot misresolve them.
- **AT-M19-5 (no drift elsewhere):** full prior unit + golden + benchmark + integration + prompt-coverage suites stay green; no `renv.lock` / renv library / study bundle / DBA run artifact is modified.
